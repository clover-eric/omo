# OMO Agent Rules

This repository implements `omo`, the OMO 边界运维管理平台.

## Required Recovery Flow

Before starting work in a new session:

1. Read `AGENTS.md`.
2. Read `docs/PROJECT_SPEC.md`, `docs/ARCHITECTURE.md`, `docs/STATUS.md`, and `docs/TASKS.md`.
3. Run `git status --short` if this directory is a Git repository.
4. Identify the current phase and acceptance criteria.
5. Make the smallest complete change for the current phase.
6. Run the relevant tests and builds.
7. Update `docs/STATUS.md` and `docs/TASKS.md`.

## Product Boundary

User-facing language must describe enterprise remote operations, boundary access, service health, diagnostics, configuration distribution, and authorized infrastructure management.

Do not implement or describe unlawful bypass, attack, stealth, third-party bulk scanning, or platform evasion workflows. Public docs, UI text, install script output, browser titles, and normal logs must avoid high-risk wording. Low-level protocol names may appear only in service library expert views, configuration generation, debug logs, and engineering documentation.

## Engineering Discipline

- Follow `MASTER_DEVELOPMENT_PROMPT.md` and the phase plan in `docs/TASKS.md`.
- Keep OpenAPI as the API contract before handler implementation.
- Do not generate core service configuration in the frontend.
- Persist long-running work through the jobs state machine.
- Keep secrets, tokens, UUIDs, and passwords generated by secure randomness.
- Do not overwrite user or other-agent changes.
- Do not push unrelated files to GitHub. Only project deployment/program files belong in commits or pushes.

## Verification

Phase 0 baseline commands:

```bash
go test ./...
pnpm --dir web build
make build
```

If a required local tool is missing, record that fact in `docs/STATUS.md` with the attempted command.
