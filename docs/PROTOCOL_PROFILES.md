# OMO Protocol Profiles

This engineering document records service profile research and must be updated when protocol capability, templates, golden tests, or subscription outputs change.

Phase 3 starts with backend-owned service profile templates in `internal/protocol`. The frontend may read these templates through `/api/services/profiles`, but it must not generate sing-box configuration directly.

Current runtime implementation: applying a service profile renders one backend-owned sing-box WebSocket entry on localhost, refreshes Caddy HTTPS routing for `/omo-access/<profile>`, starts/reloads the managed core process, and persists the generated access credential with the active service instance. Smart subscription output turns that active instance into sing-box JSON outbounds, Clash/Mihomo YAML proxies, and direct URI lines. Advanced UDP and profile-specific transport tuning remain gated by later health validation; the user-facing workflow still presents goal-based service profiles rather than raw protocol assembly.
